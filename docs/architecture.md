## コンポーネントの役割

### フロントエンド

**技術:** React 18、TypeScript、Vite、React Query、Recharts

フロントエンドはブラウザ上で動作するシングルページアプリケーションです。データの表示とユーザー入力の受け取りのみを担当します。データベース・ML モデル・予測の計算方法について直接知識を持ちません。

主な機能（予定）:
- 予測リクエストの送信フォーム（商品 ID、予測期間、任意の特徴量オーバーライド）
- 予測結果をグラフとして表示する結果ビュー
- エラーおよびローディング状態の明確な表示

フロントエンドは api-gateway とのみ通信します。REST 契約が安定している限り、バックエンドのトポロジーが変わってもフロントエンドの変更は不要です。

---

### API ゲートウェイ（Spring Boot）

**技術:** Java 21、Spring Boot 3、Spring Data JPA、Spring Validation、Flyway

ゲートウェイはすべてのクライアントリクエストに対する単一エントリポイントです。公開 API 契約を所有し、ML 推論の呼び出し前後に必要なすべての処理を担当します。

責務:
- **バリデーション** — 不正なリクエストを下流サービスに到達する前に拒否します。Bean Validation アノテーションにより宣言的かつ監査可能に保ちます。
- **永続化** — 予測リクエストとその結果を JPA 経由で PostgreSQL に書き込みます。完全な監査ログが得られ、再推論なしに後から結果を取得できます。
- **オーケストレーション** — ml-service を呼び出し、レスポンスを処理して最終的な返答を組み立てます。ml-service が遅延・不可用の場合、タイムアウト・リトライ・フォールバックを適用する場所もここです。
- **スキーマ管理** — `db/migrations/` の Flyway マイグレーションが起動時に適用され、スキーマがコードと一緒にバージョン管理されます。

ゲートウェイは ML ロジックを持ちません。ml-service を、明確に定義された HTTP インターフェースを持つ内部依存として扱います。

---

### ML サービス（FastAPI）

**技術:** Python 3.12、FastAPI、scikit-learn、pandas、numpy、Uvicorn

ML サービスはステートレスな推論エンジンです。予測リクエストを受け取り、適切なモデルを読み込み、提供された特徴量に対して推論を実行し、予測値を返します。データベースへの書き込みは行わず、フロントエンドについての知識も持ちません。

責務:
- **モデルの読み込み** — 起動時（またはモデルサイズに応じて初回リクエスト時）に、ローカルファイルシステムまたは S3 からモデル成果物を読み込みます。
- **推論** — 特徴量の前処理を適用し、モデルを実行して予測値を生成します。
- **モデル管理** — 利用可能なバージョンを列挙する `/models` エンドポイントを公開し、ゲートウェイが名前またはバージョンタグで特定のモデルを指定できるようにします。

推論を Python に集約することで、ML チームはこのサービスの外部に変更を加えることなく、モデル実装（ARIMA → LightGBM → ニューラルネットワーク）を入れ替えられます。`/docs` で自動生成される FastAPI ドキュメントは、ゲートウェイチームへのライブ契約としても機能します。

---

### PostgreSQL

**技術:** PostgreSQL 16

PostgreSQL はすべての予測アクティビティの記録システムです。api-gateway のみが直接読み書きします。

予定スキーマ（`db/migrations/` のマイグレーションファイルに従う）:

| テーブル | 目的 |
|---|---|
| `forecast_requests` | 送信された予測リクエスト 1 件につき 1 行。入力パラメータとステータスを含む |
| `forecast_results` | ml-service から返された予測値を格納。リクエストに紐づく |
| `products` | 予測リクエストから参照される SKU / 商品メタデータ |

結果をデータベースに格納することで、モデルを再実行せずに過去の予測を即座に返せるほか、監査・分析・再学習パイプラインの基盤にもなります。

---

### S3（オブジェクトストレージ）

**技術:** AWS S3 または S3 互換ストア（ローカル開発では MinIO など）

S3 は、リレーショナルデータベースや git リポジトリに置くべきでない大容量バイナリアセットに使用します。

格納内容:
- **モデル成果物** — 学習ランによって生成されたシリアライズ済みモデルファイル（`.pkl`、`.joblib`、`.onnx` 等）。ml-service が起動時にダウンロードまたはストリーミングします。
- **生データ** — 上流システムから取り込んだ CSV / Parquet ファイル。不変・プレフィックスによるバージョン管理。
- **加工済み特徴量** — 前処理パイプラインが生成した特徴量データセット。学習またはバッチ推論に使用可能な状態。

Docker ボリュームではなく S3 を使うことで、モデル成果物とデータセットをすべての環境（ローカル・ステージング・本番）から単一の場所でアクセスでき、アプリケーションコードとは独立してバージョン管理できます。

## 予測リクエストフロー

単一の同期的な予測リクエストのエンドツーエンドの流れを示します。

```
フロントエンド     api-gateway          ml-service         PostgreSQL      S3
   │                   │                     │                   │          │
   │── POST /forecast ▶│                     │                   │          │
   │                   │── INSERT request ──────────────────────▶│          │
   │                   │                     │                   │          │
   │                   │── POST /predict ───▶│                   │          │
   │                   │                     │── load model ────────────────▶│
   │                   │                     │◀─ model artifact ─────────────│
   │                   │                     │                   │          │
   │                   │                     │  推論実行          │          │
   │                   │                     │                   │          │
   │                   │◀─ 予測値 ───────────│                   │          │
   │                   │── INSERT result ───────────────────────▶│          │
   │                   │                     │                   │          │
   │◀─ 200 予測結果 ───│                     │                   │          │
   │                   │                     │                   │          │
```

**ステップごとの説明:**

1. ユーザーが予測フォームに入力して送信します。フロントエンドは商品 ID・予測期間・任意の特徴量オーバーライドを含む `POST /api/v1/forecast` を api-gateway に送ります。

2. ゲートウェイがリクエストペイロードを検証します。不正なリクエストは下流への呼び出し前に `400` で拒否されます。

3. ゲートウェイは PostgreSQL にステータス `PENDING` の `forecast_request` レコードを保存します。

4. ゲートウェイは関連する特徴量を渡して ml-service の `POST /predict` を呼び出します。

5. ml-service は S3（またはウォームなインメモリキャッシュ）からモデル成果物を読み込み、推論を実行して予測配列を返します。

6. ゲートウェイは `forecast_request` のステータスを `COMPLETE` に更新し、`forecast_result` レコードを PostgreSQL に書き込みます。

7. ゲートウェイは HTTP レスポンスで予測結果をフロントエンドに返します。

8. フロントエンドが結果を時系列グラフとして描画します。

## 関心の分離

各サービスは明確に区切られた責務を持ちます。以下の表は関心事とその所有者のマッピングです。

| 関心事 | 所有者 | 理由 |
|---|---|---|
| UI とデータ表示 | `frontend` | UI の変更でバックエンドの再デプロイが不要 |
| API 契約と入力バリデーション | `api-gateway` | 全クライアントへの単一の安定したインターフェース |
| ビジネスデータの永続化 | `api-gateway` → PostgreSQL | トランザクショナルな書き込みはドメインモデルの近くに置く |
| ML 推論ロジック | `ml-service` | Python ML エコシステムを JVM から隔離する |
| モデル成果物の保存 | S3 | バイナリアセットをコードと DB の両方から切り離す |
| スキーマバージョニング | `db/migrations/`（Flyway） | スキーマ変更をコードレビューし、決定論的に適用する |

**すべてを単一サービスに入れない理由:**

単一サービスのアプローチは開始当初はシンプルですが、プロジェクトの成長とともに高コストな結合を生み出します。

- 推論と REST API がプロセスを共有すると、大きなモデル読み込みによるメモリスパイクが他のすべてのリクエストの API レイテンシに影響します。
- ML モデルの更新がアプリケーション全体の再デプロイを必要とします（フロントエンド向け API を含む）。
- Java と Python の ML ライブラリは、Jython・GraalPy などの不便なブリッジレイヤーなしに同一ランタイムで使用できません。結果は遅く、保守困難で、Python ML エコシステムから切り離されます。

**このアーキテクチャのコスト**は、複数サービスの運用オーバーヘッドと、ゲートウェイと ML サービス間の内部 HTTP 呼び出しです。ミリ秒単位のレイテンシが求められない需要予測ワークロードにおいては、本番 ML システムで広く採用される許容可能かつ標準的なトレードオフです。

# AWS デプロイ方針

このドキュメントは、当プロジェクトを AWS 上に最小構成でデプロイする場合の方針をまとめたものです。

**位置づけ:** 最小可行デプロイ計画です。生産環境向けの最終アーキテクチャではありません。

---

## 1. 目的

当プロジェクトは現在、Docker Compose によるローカル実行を前提として設計・検証されています。このドキュメントでは、現在のシステムを AWS 上に**最小限の複雑さで**デプロイするとしたらどうするかを整理します。

目標は「実現可能で説明できる最小構成」です。ECS や RDS、ALB といった AWS サービスを使いこなすことより、**構成の意図と trade-off を明確にすること**を優先します。

---

## 2. 最小構成（Minimum Viable Deployment）

```
[ユーザー]
    ↓
[EC2 インスタンス（1台）]
├── Frontend（React + Vite build → nginx または vite preview）
├── API Gateway（Spring Boot :8080）
├── ML Service（FastAPI :8000）
└── PostgreSQL（:5432）
    ↓
[S3]
├── demand-forecast/models/   ← model.pkl / artifacts.pkl
└── demand-forecast/raw/      ← データセット（将来用）
```

**選択の理由:**
- 既存の Docker Compose 構成をほぼそのまま EC2 上で動かせる
- 新たな AWS サービスを最小化することで、セットアップコストと障害箇所を抑える

---

## 3. 各 AWS サービスの役割

| サービス | 役割 |
|---|---|
| **EC2** | アプリケーション実行基盤。Docker Compose で全サービスを起動する |
| **S3** | 学習済みモデル成果物（`model.pkl` / `artifacts.pkl`）とデータセットの保管 |

**この構成に含めないもの（将来拡張）:**

| サービス | 理由 |
|---|---|
| RDS | PostgreSQL を EC2 内に同居させることで、初期セットアップを単純化する |
| ECS / Fargate | コンテナオーケストレーションは最小構成に不要 |
| ALB | 単一 EC2 への直接アクセスで十分。スケーリングが必要になったタイミングで導入する |

---

## 4. デプロイ手順の概要

以下は構成の流れであり、詳細な操作手順書ではありません。

```
1. EC2 インスタンスを起動（Amazon Linux 2023 推奨）
2. Docker / Docker Compose をインストール
3. リポジトリを git clone
4. 環境変数を設定（.env ファイルを作成）
5. docker compose up --build
6. アプリの疎通確認（frontend, API Gateway, ML Service）
7. S3 モードを使う場合は STORAGE_MODE=s3 を設定し、
   upload_artifacts_to_s3.py でモデルをアップロード済みにしておく
```

EC2 のセキュリティグループでは、少なくとも以下のポートを開放します：

| ポート | 用途 |
|---|---|
| 3000 | Frontend |
| 8080 | API Gateway |
| 8000 | ML Service（通常は内部通信のみ。デバッグ時のみ開放） |

---

## 5. 環境変数 / 設定の考え方

`.env.example` に定義されている変数をベースに、EC2 上で `.env` ファイルを作成します。

**ストレージモードの選択:**

| STORAGE_MODE | 動作 |
|---|---|
| `local`（デフォルト） | コンテナ内の `ml-service/models/` からモデルを読み込む |
| `s3` | 起動時に S3 からモデルをダウンロードしてキャッシュする |

EC2 に S3 アクセス権を持つ **IAM ロール**をアタッチすることで、アクセスキーをコードや環境変数に直接埋め込まずに済みます。

**設定してはいけないこと:**
- AWS アクセスキーをコードやドキュメントに記載しない
- `.env` ファイルを Git にコミットしない

---

## 6. この構成を選んだ理由（Trade-offs）

### メリット

| 観点 | 内容 |
|---|---|
| 実装コスト | 既存の Docker Compose 構成をほぼ変更せずに転用できる |
| セットアップ時間 | 慣れた構成を使うため、初期立ち上げが速い |

### デメリット

| 観点 | 内容 |
|---|---|
| 単一障害点 | EC2 1台のため、障害時にすべてのサービスが停止する |
| DB とアプリの同居 | PostgreSQL が EC2 上にあるため、インスタンス障害でデータリスクがある |
| スケーラビリティ | 水平スケールが難しく、トラフィック増加に対応しにくい |
| 運用負担 | OS・Docker・アプリの更新を自前で管理する必要がある |

**結論:** この構成は、システムの動作を証明し説明するための**最小可行デプロイパス**です。本番運用を想定した場合は次のステップへの移行が前提となります。

---

## 7. 将来的な拡張

現時点ではスコープ外ですが、本番化に向けた自然な拡張方向は以下の通りです。

| テーマ | 内容 |
|---|---|
| PostgreSQL → RDS | データ永続化をマネージドサービスに移管 |
| 単一 EC2 → ECS / Fargate | コンテナをサービス単位でスケーリング可能にする |
| ALB + Route 53 | ドメイン・HTTPS・ロードバランシングの導入 |
| CI/CD | GitHub Actions などを使ったデプロイ自動化 |
| Secrets Manager | 環境変数・認証情報の安全な管理 |

これらは現在の PoC 段階では不要ですが、段階的に導入することでシステムを本番品質に引き上げられます。
